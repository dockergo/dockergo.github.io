@eco
事务与并发控制要同时满足两个目标：**隔离性**（并发事务互不干扰，效果等价于某个串行序）与**性能**（尽量让读写并行）。上图是这一类系统的通用生态：时间戳/版本给操作定序，锁或校验解决写冲突，读走快照、写靠仲裁点原子成败——四类算法在「悲观 vs 乐观」「有无多版本」的象限里各占一角。

@m1
MVCC 让读写不再互相加锁：写不覆盖旧值而追加带 `⟨写事务, commit-ts⟩` 的新版本，读在自己的快照里挑「可见的最新版本」。**可见性判定**是心脏：读事务持 `start-ts` + 活跃集，对某版本 v（写者 w），只有「w 已提交 且 `commit-ts ≤ start-ts` 且 w 不在活跃集」才可见，从新版向旧遍历命中第一个可见即停。由此得三个好处——**读不阻塞写**、**可重复读**（快照点固定）、清晰的 **GC 边界**（比最老活跃 start-ts 更旧且被覆盖的版本可回收）。

@m2
纯快照隔离（SI）有一个漏洞：**写偏（write skew）**——两个事务各读一份重叠数据、各写不同 key，单独看都合法，并发却破坏了跨行约束，SI 检测不到。**SSI（可串行化快照隔离）**在 SI 之上追踪事务间的 **rw-反依赖**（T1 读了某数据、T2 随后写它），当出现「两条相邻反依赖构成的危险结构」——理论已证这是不可串行化的必要征兆——就中止其中一方。代价是少量误杀（保守中止本可串行的事务），换来真正的可串行化，且仍保留 SI「读不阻塞写」的优点。

@l1
2PL（两阶段锁）是悲观路线：执行期就加锁——读加共享锁、写加排他锁，且遵守**两阶段**纪律（增长期只加不放、收缩期只放不加），由此保证冲突可串行化。冲突表现为**等锁**，于是可能出现**死锁**（事务互等对方持有的锁形成环），靠**等待图检测**（发现环则中止一方）或**超时**兜底。幻读需**谓词锁 / 间隙锁**锁住「尚不存在的行」的区间，防并发插入。2PL 在高冲突下比乐观路线稳（不重试、只阻塞），代价是读写互阻、并发度受锁粒度限制。

@o1
OCC 走「**先改后验**」的乐观路线，全程不加锁，赌冲突罕见。三阶段：**读阶段**记下每个 key 的版本号（read set），写只进本地缓冲；**验证阶段**（原子临界区）逐个检查 read set 的 key 当前版本是否仍等于当初读到的——全等则 COMMIT，任一不等则 ABORT；**写阶段**验证通过才落库。正确性两条铁律：**验证与写生效必须原子**，且验证要覆盖完整 read set（含范围读的幻影检查）。低冲突场景最优（无锁无死锁），高冲突时 abort 重试浪费急剧放大。

@d1
分布式事务跨多分片、没有全局锁。Percolator 的巧思：从要写的 key 里**选一个当 Primary**，其余 Secondary 锁都存「指向 Primary」的引用，于是「事务是否提交」被归约成「Primary 锁是否已换成 commit 记录」这**一个单 key 原子事实**。**Prewrite**：取 start-ts、给每个写 key 上锁并写未提交版本，遇冲突回滚；**Commit**：取 commit-ts、**先原子提交 Primary**（事务即算成功），Secondary 异步跟进。兜底靠**读者 lazy 清锁**：读到残留 Secondary 锁就顺引用查 Primary，已提交则替它补提交、超时则回滚。这把经典 2PC 的「协调者阻塞点」下放成一个任何人都能探查的 key，故障恢复不再依赖协调者日志。

@cmp
四类并发控制在同一目标（可串行化 / 隔离）下的取舍，本质是两个正交维度的四象限：**冲突在执行期挡（悲观 2PL）还是提交期验（乐观 OCC）** × **是否维护多版本（MVCC 有 / 纯锁无）**。2PL 高冲突稳但读写互阻；MVCC-SI 读写不互阻、读多最优、写偏需 SSI 补；OCC 低冲突最优、高冲突重试爆炸；Percolator/2PC 专解跨分片原子提交。没有普适最优——按读写比、冲突率、是否跨分片选型。

@eng
MVCC 的快照读内核几乎人人相同，工程差异集中在三处：**旧版本存哪、谁来 GC、要不要跨分片原子提交**。**PostgreSQL** 把多版本直接留在堆页（新版本另存、旧版打 `t_xmax`），可见性靠快照 `xmin/xmax/xip` 对照行头，死版本由 **VACUUM/autovacuum** 回收并冻结老事务防 XID 回卷，默认隔离级别是 Read Committed；**MySQL InnoDB** 走另一路——就地更新、旧值进 **undo 回滚段**串成版本链，读时按 **ReadView** 沿 `DB_ROLL_PTR` 回溯，旧版本由 **purge 线程**清理，默认是 Repeatable Read（间隙锁防幻读）。两者都是单机，2PC 只用于外部 XA / 逻辑复制。

跨节点场景把这套搬进**带时间戳的 KV**：**TiKV** 用 **Percolator** 模型（RocksDB 三列族 `CF_DEFAULT/CF_LOCK/CF_WRITE` = data/lock/write + 全局 `start_ts`/`commit_ts`），提交走两阶段——`Prewrite` 给每行上锁并选一个 key 作 **Primary lock**，`Commit` 先原子提交 Primary 即代表事务成败、Secondary 异步跟进，兜底靠读者顺 Primary 引用清锁；**CockroachDB** 用 **HLC 时间戳** + 事务记录，默认即 **Serializable**（乐观 + timestamp cache + 读刷新 refresh），提交用 **Parallel Commit**（意图全就位即判提交）省一轮 RTT。一句话：**单机把可见性做进堆 / undo，分布式把它做进带 ts 的 KV 并用 2PC / 并行提交收口**——版本存储位置与提交是否跨节点，是这条线的两根取舍主轴。权威参考：[PostgreSQL MVCC 官方文档](https://www.postgresql.org/docs/current/mvcc.html)、[InnoDB Multi-Versioning 官方文档](https://dev.mysql.com/doc/refman/8.0/en/innodb-multi-versioning.html)、[TiKV Deep Dive · Percolator](https://tikv.org/deep-dive/distributed-transaction/percolator/)、[CockroachDB 架构 · Transaction Layer](https://www.cockroachlabs.com/docs/stable/architecture/transaction-layer)。
