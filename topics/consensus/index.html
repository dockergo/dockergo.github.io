分布式共识要解决的根本矛盾：会各自宕机、消息会丢会乱序的多台机器，如何对「发生了什么、以什么顺序发生」达成一致。答案是复制状态机——把写变成只增不改的日志，选出唯一 Leader 作序列化点复制给副本，条目被多数派确认即永久提交，各副本按同序回放收敛到同一状态。安全性来自「多数派相交」（任两多数派必有公共成员，已提交条目跨换届不丢），活性来自选举（Leader 失效则 term+1 重新选主）。

R1Raft · Leader 选举与任期

任期 term 单调递增，每个 term 至多一个 Leader（选举安全）。Follower 选举超时未收到心跳即自增 term、转 Candidate 拉票，收到含自身的多数派选票即当选。随机化选举超时错峰、打破对称，避免选票瓜分导致的空转 term。只投票给「日志至少和自己一样新」的候选人，保证新 Leader 含全部已提交条目。

R2Raft · 日志连续性回溯匹配

日志匹配性质：若两副本在同一 index 处 ⟨term,idx⟩ 相同，则其之前所有条目必然全同。故 Follower 落后或尾部分叉时无需逐条比对，只要找到最后一致点、其后一律以 Leader 为准覆盖。Leader 靠 AppendEntries 的 prevLog⟨idx,term⟩ 试探，不匹配就回退 nextIndex 重试直到命中，再单向前进对齐。关键不变式：已提交条目永不被覆盖，覆盖只发生在未提交的分叉尾部。

R3Raft · 成员变更（联合/单步）

增删节点的风险是新旧配置的多数派若不相交，可能同 term 选出两个 Leader（脑裂）。联合共识 C_old,new 让过渡期的每个决议同时满足新旧两个多数派，把两配置绑定叠加，跨越安全点后再切到 C_new。单节点变更每次只增减一个成员，相邻配置的多数派天然相交，无需过渡配置，实现更简单（主流选择）。新增节点先作 learner 追平日志、不计票，变更本身作为一条特殊日志条目复制提交。

R4Raft · ReadIndex / Lease 读一致

把读也走日志复制太贵；线性一致读只需两点保证：读时刻确有最新 commit 视图、且本地状态机已 apply 到该点——无需产生新日志。ReadIndex：记下 commitIndex，广播一轮心跳收多数派响应以证明「此刻我仍是 Leader」，等 applied ≥ readIndex 后读本地快照。Lease 用短于选举超时的时钟租约免掉每读的心跳 RTT，代价是依赖有界时钟漂移，故稳态用 lease、时钟不可信时回退 ReadIndex。两者共同的 applied 屏障不可省。

P1Multi-Paxos · 两阶段 prepare/accept

Basic Paxos 决议单个值分两阶段。Phase 1 Prepare(n)：以全局唯一递增的提案号 n 抢占，拿到多数派「不再接受更小编号」的承诺，并回收各 Acceptor 已接受的值。Phase 2 Accept(n,v)：在承诺范围内提交值，多数派 Accepted 即 chosen。安全性关键：若 Phase 1 回收到已接受值，Proposer 必须沿用编号最大的那个值而非自己的——因此一旦某值被多数派选定，后续更高编号提案只会再次选定同一值，值唯一且不可变。

P2Multi-Paxos · 稳定 Leader 摊销一阶段

Basic Paxos 每决议两轮 RTT，且并发 Proposer 互相抬高编号会活锁。Multi-Paxos：选出稳定 Leader，用一次 Prepare(n) 对「从某 slot 起的所有未来实例」预承诺，此后每条命令只跑 Phase 2——稳态每决议仅 1 轮 RTT。这不违反单决议安全性：每个 slot 仍是独立 Paxos 实例，编号 n 在这些 slot 上持续有效。Leader 更替相当于用更大 n' 重跑一次 Phase 1 回收+补洞，故障切换偶发一次一阶段、稳态零一阶段。

Z1ZAB · 原子广播提交

ZAB 是原子广播而非日志复制变体。Leader 把每个写封成带 zxid 的事务提案：广播 PROPOSE，follower 落盘回 ACK，收到过半 ACK 后广播 COMMIT，各副本按 zxid 升序 apply。zxid = ⟨epoch, counter⟩：epoch 标识 Leader 届次、counter 届内单调递增；FIFO 有序信道 + 单调 counter 是全序的物理基础。保证全序（任两消息在所有副本先后一致）与因果前缀（提交是连续前缀、不留空洞）。

Z2ZAB · 崩溃恢复同步

Leader 崩溃后先选 zxid 最大的副本当新 Leader——它必含所有已 COMMIT 提案（提交需过半、任两过半集合相交）。新 Leader 递增 epoch，再用三种原语把 follower 同步到统一前缀：DIFF 补发缺失提案、TRUNC 截断未提交的悬挂提案、SNAP 差距过大时直传快照。恢复不变式：已提交提案必保留、未提交悬挂提案一律丢弃，epoch 递增使旧届残留无法与新 zxid 混淆，所有副本收敛到同一前缀后恢复广播。

⇄Consensus & Replication · 算法核心差异对比

三者同解「就一条全序日志达成一致」，都用「唯一 Leader + 过半 quorum + 单调届次」骨架，分歧在五个维度：序列化点（Raft 强 Leader 单向流动／Paxos Leader 是可选优化／ZAB 原子广播）、选主（Raft 内建 term 投票／Paxos 靠提案号抢占、选主外挂／ZAB 选 zxid 最大者）、日志模型（Raft 连续无空洞／Paxos 按 slot 独立可空洞并行／ZAB zxid 前缀提交）、成员变更、恢复方式（Raft 日志回溯／Paxos 重跑 Phase 1／ZAB 显式 DIFF/TRUNC/SNAP）。内核同构：term ≈ 提案号 ≈ zxid.epoch，log index ≈ slot ≈ zxid.counter，多数派相交是共同的安全性基石。

⚙Consensus & Replication · 工程实现差异（真实项目落地取舍）

同一套 Raft，落到真实系统里被封装成完全不同的形态——差异不在协议，而在边界画在哪、I/O 谁来做。etcd 把 Raft 内嵌进进程、配上 MVCC KV 与 gRPC，交付开箱即用的 CP 存储服务，WAL 与 boltdb 都自管；hashicorp-raft 反其道做成「电池全含」的可嵌入库，宿主 import 后注入 FSM + LogStore/StableStore/SnapshotStore + Transport 六件套，库只持共识状态机、存储与传输可插拔；etcd-raft 更极端，只留纯状态机核（Step→Ready），连驱动复制的线程都不起（RawNode 明言非线程安全），所有磁盘与网络 I/O 由宿主的 Node 循环外包——正因如此它能被 etcd 与 CockroachDB 同时复用。读一致也各有取舍：etcd / etcd-raft 用 ReadIndex（一轮心跳确认领导权、不写日志），hashicorp-raft 额外提供 Leader Lease 免掉每读一次 RTT。

ZooKeeper 则不走 Raft，而用 ZAB 原子广播：写经唯一 Leader 定序（zxid = ⟨epoch, counter⟩），读默认落本地副本、可能略旧（sync() 强制追平），崩溃恢复走独立的「发现新 epoch + DIFF/TRUNC/SNAP 同步」阶段——这是它区别于 Raft「恢复即复制特例」的定义性设计。一句话：算法同构，工程封装层级（服务 / 库 / 内核）与 I/O 归属（自管 / 注入 / 外包）才是选型的真实分水岭。权威参考：etcd 官方文档 · Learning、etcd-io/raft 源码、hashicorp/raft 源码、ZooKeeper Internals（ZAB）。
