@eco
查询引擎要跨越一道鸿沟：用户写的是**声明式**的 what（「我要这些数据」），机器要的是**过程式**的 how（按什么顺序、用什么算法、跑什么指令）。分工很清晰——**优化器决定『做什么形状』，执行器决定『多快做完』**，两者以一棵物理执行计划为界。前半是搜索问题：同一查询有无数结果等价、代价迥异的计划，优化器靠代价模型 + 统计信息选出中间结果最小的那个；后半是效率问题：拿到正确计划后，逐行解释与向量化 / JIT 之间可差一个数量级。省的地方不同——优化器省「跑错计划」的巨额浪费（一次性、离线味），执行器省「每一行的解释开销」（每行、在线味）。

@q1
连接顺序是优化器最硬的骨头：n 表连接的顺序数随 n **阶乘级**爆炸，无法全枚举。但连接有**最优子结构**——一个表集合的最优计划，可由其子集的最优计划加上连接代价拼出——于是优化器**自底向上填表**：`size 1` 单表访问是基例，逐层拼到 `size n` 全集。关键在于每个子集的最优代价**只算一次并缓存**、被更大子集复用，把阶乘级压到 `2ⁿ` 量级。代价 = `估行数 × 单位代价`，估行数来自统计信息，故「最优」准不准最终取决于统计准不准；表太多时 `2ⁿ` 仍爆，退化为左深树 / 贪心等启发式。

@q2
执行器的第一个效率杀手是**逐行的火山模型**：每算子每行付一次虚函数调用与类型分支，真正计算只占少数，大半 CPU 花在「解释」上。**向量化**把粒度从「一行」拓宽到「一列批」（如 1024 行的若干列），固定的调用 / 分派开销被**批大小摊薄**，紧凑列数组带来缓存友好与自动 **SIMD** 化。SIMD 是第二层加速：一条指令对一列相邻多值**并行**运算（如过滤 `值 > 8`），产出选择位掩码驱动下游，全程无逐行 `if`。收益来自摊薄解释 + 数据并行 + 缓存友好三者叠加，代价是批增大延迟粒度、物化中间列占内存。

@q3
向量化拓宽每次处理的**数据**，**JIT** 则压缩每次处理的**指令**。逐行解释表达式 `a*2+b` 时，每行都重走一遍表达式树（递归 `eval` + 按节点类型分支 / 虚调用），开销乘以行数即主要成本。JIT 在查询开始时把这棵「对每行都一样」的树 `树 → IR → 机器码` 一次性编译成专用原生指令，之后每行直跑、无遍历无分支。核心权衡是**编译有固定成本、靠执行次数摊销**（编译一次执行万次才划算）；JIT 与向量化正交，现代引擎常组合成向量化 + JIT。

@cmp
火山迭代器、向量化、编译(JIT) 拿到的是**同一份正确物理计划**，同解一个病灶——逐行「解释开销」，分歧只在**处理粒度**：一次一行 / 一次一列批 / 整条流水线段融成一段代码。**火山**不治此病（作最朴素基准，实现最简但每行都重缴解释税，数据一大就最慢）；**向量化**「拓宽每次处理的数据」，把调用开销 ÷ 批大小、再叠加 SIMD 数据并行与缓存友好；**编译**「压缩每次处理的指令」，把类型定死、流水线展开成原生机器码，从根上消除解释。取舍随场景：向量化是 OLAP 扫描 / 聚合的主力，编译须「编译一次、执行万次」摊销固定成本，二者正交、现代引擎常组合使用。

@eng
四个引擎共享「声明式 → 优化器搜代价最小计划 → 执行器榨 CPU」的同一骨架，取舍分野在三条轴上。**DuckDB** 进程内嵌入、**push-based 向量化**流水线（DataChunk 2048 行列批），靠 buffer manager 做 out-of-core 溢写，专攻单机分析。**ClickHouse** 用 **Processors 拉取执行图** + 大 Block（约 6.5 万行）、优化以规则为主，主打实时列式 OLAP 与大宽表聚合。**Spark** 用 Catalyst（规则 + CBO + AQE）选计划、Tungsten **whole-stage codegen** 把一串算子编成 JVM 字节码、shuffle **物化到磁盘**换容错，胜在大数据批与 ETL / ML。**Trino** 是纯 **MPP**：Driver 泵拉动算子链、Page 在内存流水线里跨 worker 交换、默认**全内存不落盘**（仅容错执行 FTE 才 spool），为跨数据源的交互式联邦查询而生。**为什么不同**：越往分布式端，容错与落盘物化越重（Spark / Trino）；越往单机端，向量化流水线越极致（DuckDB / ClickHouse）；而 codegen（Spark）与解释式向量化（DuckDB / ClickHouse / Trino）则是「编译一次消除虚调用」与「按批摊薄解释开销」两条榨 CPU 的路线。
