存储引擎要回答一个物理问题：数据以什么形状躺在磁盘 / SSD 上，才能写得快、读得动、又不把空间撑爆。地基是介质特性——顺序 IO 快、随机 IO 慢，SSD 还以块擦写、就地改贵。于是引擎普遍收敛到：写先入内存缓冲 + 落 WAL 保命 → 满则顺序刷成不可变有序单元 → 后台整理合并重写 → 读跨内存与多个落盘单元归并。三种核心机制（LSM 追加、B-link 就地并发、列存布局）在这套骨架里各解一面。

S1LSM-tree · Compaction 写放大与 L0 停顿

LSM 是「写优先」的极端：写只进内存有序表，满了整块顺序刷成 L0 文件——写路径几乎无随机 IO，写吞吐极高。代价分两笔。读放大：L0 文件键区间可重叠，查一个 key 可能翻遍所有 L0；后台 compaction 把上层读出、与下层归并、重写成层内不重叠的更大文件下沉。写放大：一条数据从 L0 沉到 Ln 被逐层重写约「层数」次。L0 停顿是结算点：前台写速持续超过 compaction 消化速度、L0 堆积越过阈值就限速甚至完全停写——它不是 bug，而是「不让读放大失控」的反压闸门。

S2B-link tree · 无锁页分裂（兄弟指针）

B+ 树读优先、点查稳定，但有并发死穴——页分裂：满页一分为二时，普通做法要锁父锁子，否则并发读可能顺「即将失效」的父指针走到正被搬空的页。B-link tree 用结构冗余化解：每节点加 high key（键上界）+ 指向右兄弟的指针。分裂拆成两个局部原子步——先建新兄弟、搬右半键、接好兄弟链、设 high key（此刻树已可达且正确，哪怕父指针未更新），再把分隔键升入父节点。并发读发现 查找键 > high key 就知目标已右移、顺兄弟指针右移即命中。high key 自愈过期父指针，父更新可安心延后，代价仅偶尔多跳一兄弟——换来读与分裂互不上锁。

S3列存 / 向量 · 压缩管线 + SIMD 下推

列存换一个维度：把同一列的值连续存放，同时解锁压缩与计算两条增益。压缩：同列同类型高度同质，编码链层层级联——RLE 吃连续重复、字典编码 把宽值映射成小整数、bit-pack 压到比特级，数值列还有 delta / frame-of-reference，压得越狠扫描搬的字节越少。计算：定长编码可在编码域直接比较、对齐 SIMD 寄存器宽度，查询把谓词下推做两级跳过——先用列块 min/max 整块跳过，再对命中块解码后用 SIMD 批量比较产出无分支的选择位掩码。代价是不擅长「取整行所有列」（跨列拼装），故服务 OLAP 扫描聚合而非点查点改。

⇄Storage Engine · 算法核心差异对比

LSM 与 B-tree 是同一份数据的两种物理布局，本质是在 RUM 三角（读放大 / 写放大 / 空间放大，三者不可兼得）上选不同工作点：LSM 用「推迟合并」换低写放大（写多首选），代价是读归并多层、旧版占空间；B-tree 用「就地维护有序」换低读放大（读多 / 点查首选），代价是随机写与页半满。列存是与之正交的第三维——不改「LSM vs B-tree」的选择，而是把「一行的所有列」换成「一列的所有行」，服务扫描聚合。没有普适最优，只有对负载的适配。

⚙Storage Engine · 工程实现差异（真实项目落地取舍）

把「同一存储难题」放到四个真实系统里看，差异恰好落在 RUM 三角的四个工作点上。RocksDB 走 LSM——写只追加、后台 compaction 逐层归并，以写放大换极低写延迟，是高写入点查 KV 引擎的底盘。PostgreSQL 走 8KB 堆页 + B树索引 + 就地更新，旧版本由 MVCC 保留、VACUUM 回收，读放大低、点查稳定，服务 OLTP。ORC 是列存文件：一列的所有行连成 stream，用 RLE / 字典编码，配 stripe 级统计与谓词下推，整文件不可变、一次写多次批量读，专供 OLAP 大扫。Iceberg 再抽象一层——它不碰数据文件字节，只用「快照 + 清单」这棵元数据树给底层列存文件（Parquet / ORC）叠加 ACID、时间旅行与分区裁剪；行级删除也靠独立 delete file 做 merge-on-read，而非重写数据。为什么不同：介质顺序快、随机慢是共同地基，取舍全在负载的读写比与可变性需求——越偏点写就越保留原地可变（RocksDB / PostgreSQL），越偏海量批量分析就越走向不可变、元数据越重（ORC / Iceberg）。
