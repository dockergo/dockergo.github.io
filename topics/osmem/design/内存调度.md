@eco
操作系统内存管理同时回答两件事：**每个进程如何拥有一片看起来连续、又与他人隔离的地址空间**，以及**有限的物理内存如何在众多进程间高效供给且不被某一方吃垮**。前者靠地址虚拟化——`页表`记录虚拟页到物理帧的映射、`TLB` 缓存热映射做加速，真正访问到才建立映射（缺页）。后者靠物理供给与隔离——`伙伴系统`按 `2^order` 管大块页、`Slab` 在其上做小对象池，`cgroup` 再用 **high / max 两道水位线**给每组进程画出内存边界。一句话：页表 + TLB 负责「地址怎么翻译」，分配器 + cgroup 负责「物理页怎么发、发多少」。

@v1
地址翻译按命中概率分三档，绝大多数访存要极快。CPU 发出虚拟地址后 MMU 先查 **`TLB`**：命中即一步拿到物理帧、纯硬件无内核介入。未命中走 **页表 walk**：硬件逐级索引多级页表读出 `PTE`，有效则取帧号并**回填 TLB**。若 PTE 无效则触发**缺页异常**陷入内核——minor fault 只需补建映射，major fault 须发起磁盘 I/O；内核填好 PTE 后重执触发指令，故障对程序完全透明。优化本质就是提高命中率、少触发缺页。

@v2
物理内存分配面对两种尺度，内核分了两层。底层 **伙伴系统（buddy）**以页为单位、按 `2^order` 组织空闲块：请求向上取整到最近 order，缺块则从更大 order **逐级拆半**，释放时检查「伙伴块」是否也空闲、若是则**合并**上移，以此回收**外部碎片**。上层 **Slab / Slub** 面向内核海量同类小结构体（`inode`、`task` 等）：向 buddy 要整页、切成**等长对象槽**做对象池，分配 / 释放都是 O(1) 出入池，解**内部碎片**并复用已初始化的热对象。粗粒度靠 buddy、细粒度靠 slab，是内核分配的骨架。

@v3
光能分配还不够——多组进程共享物理内存时须防止某组把内存吃光拖垮全机，`cgroup` 用**两道水位线**让回收压力逐级升级，构成状态机。用量低于 **`memory.high`（软限）**时分配畅通；越过即进入**节流 + 主动回收**（丢干净页缓存、换出脏页），此时不杀进程。继续涨到 **`memory.max`（硬限）**则**阻塞新分配**并同步强回收，兜底才触发 **OOM killer** 选进程杀掉。整条链是**先软后硬、先回收后杀**，且回收 / OOM 只发生在各组自己的笼子里，绝不波及同机其他组。

@cmp
两组机制各有取舍。**buddy vs slab**：同一分配链的上下两层——buddy 以 `2^order` 管**大块页**、合并伙伴块解**外部碎片**，slab 在整页之上切**等长对象槽**、O(1) 出入池解**内部碎片**，且 slab 的页正来自 buddy。**cgroup v1 vs v2**：v1 各控制器**独立挂载、层级割裂**，内存软硬限分散、语义弱；v2 **统一单一层级**，引入 `memory.high` / `memory.max` **分级水位**与 PSI 压力反馈，是当下容器运行时首选。前一组是分工（尺度 / 碎片类型），后一组是代际演进（层级模型 / 水位语义）。

@eng
把「用户态运行时如何在内核内存之上分层」并排看，会发现三者**惊人同构**：都是「**每线程 / 每 P 本地缓存做无锁快路径 + 全局慢路径 + 分级供给**」——差异全落在**回收**与**隔离**两处。**Linux 内核**是底座本身：`buddy` 按 `2^order` 管大块页、`slab/slub` 在其上切等长对象槽（per-CPU 缓存做无锁快路径），回收靠**手动 `free`/`kfree`** 加页回收，隔离靠 `cgroup` 的 high/max 水位线兜到 OOM killer。**Go 运行时**在 OS 之上（向内核 `mmap` 批发 arena）再叠一层源自 tcmalloc 的 `mheap → mcentral → mcache`：`mcache` 每 P 一个、无锁，小对象按约 68 档 `size class` 向上取整切 `mspan` 槽位（外加 tiny 分配器合并极小对象）；回收是**并发三色标记清除、非分代、非压缩**（省停顿、但不移动对象消外碎片），隔离靠 `GOMEMLIMIT` 软限 + `GOGC` 步调。**JVM（HotSpot）**同样在托管堆之上给每线程一段 **TLAB** 做指针 bump 无锁分配，但回收是**可插拔 GC**（分代 G1 用 Region + 写屏障/RSet、ZGC 用着色指针 + 读屏障并发重定位），且多数收集器会**压缩 / 疏散搬活对象消碎片**，隔离靠 `-Xmx` 硬上限（超限即 `OutOfMemoryError`）。

一句话：**本地缓存快路径人人都做，分野在回收（手动 vs 并发 GC vs 可插拔 GC）与隔离（cgroup 关笼子 vs runtime 软/硬限额）**。旁注 **Rust** 走第四条路——无 GC、无运行时堆管理器，靠所有权 + `Drop` 在**编译期**定回收点，默认用系统 `malloc`（可换 jemalloc），把「何时回收」从运行期挪到编译期。Go / JVM / Linux 的细节已在本库 [Go 内存分配器](../../projects/go/)、[OpenJDK 可插拔 GC](../../projects/openjdk/)、[Linux 虚拟内存](../../projects/linux/) 中源码核实。权威依据：[Go GC Guide](https://go.dev/doc/gc-guide)、[HotSpot GC Tuning Guide](https://docs.oracle.com/en/java/javase/21/gctuning/introduction-garbage-collection-tuning.html)、[Linux MM docs](https://docs.kernel.org/admin-guide/mm/index.html)、[tcmalloc](https://google.github.io/tcmalloc/)。
