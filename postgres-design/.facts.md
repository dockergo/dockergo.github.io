# PostgreSQL 源码核实事实台账（source-verified fact ledger）

> 源码：`~/workdir/postgres`（内核在 `src/backend/`）。每条事实从真实源码 grep/read 核实,标 `file:line`。
> 判型:**进程级关系数据库内核**(process-per-connection RDBMS kernel) —— postmaster fork-per-connection + 共享内存协调 + WAL 预写日志(持久) + MVCC 多版本 + 代价优化器 + Volcano 拉取执行器,一体在单机磁盘存储引擎上。
> 接触面:客户端 libpq 连接 → 每连接一个 backend 进程;后台辅助进程(checkpointer/bgwriter/walwriter/autovacuum/startup/walsender/walreceiver)围绕共享内存协作。
> 设计张力:**MVCC 无锁读(读不阻塞写)vs 膨胀(死元组需 VACUUM 回收)** —— 旧版本原地保留保证快照读,代价是 bloat + 后台清理。并存:**WAL 持久性(commit fsync)vs 写吞吐**、**进程隔离/稳健 vs fork 连接开销(故需连接池)**。

## 判型依据(进程模型 + 共享内存边界)
- `postmaster/postmaster.c:497 PostmasterMain` —— 主守护进程入口;`:1678 ServerLoop` 事件循环。
- `:1728 BackendStartup` —— 每个 `WL_SOCKET_ACCEPT` 到来 `AcceptConnection` 后 fork 一个后端。
- `:3627 postmaster_child_launch(...)` —— 实际 fork/exec 子进程(fork-per-connection 判型铁证)。
- `storage/ipc/ipci.c:120 CreateSharedMemoryAndSemaphores` —— 所有进程共享一块 shmem(缓冲池/WAL buffer/锁表/ProcArray),这是"进程模型"而非"线程模型"的协调基座。

## 主线 1 · 进程模型(postmaster fork · Control Path)
- `postmaster/postmaster.c:497 PostmasterMain` 启动 → `:1678 ServerLoop` 等待事件 → `:1728 BackendStartup(&s)` 每连接 fork 后端。
- `:3576 BackendStartup` / `:4010 StartChildProcess` / `:3627 postmaster_child_launch` —— 子进程创建机制。
- 后台辅助进程(postmaster 拉起,各有独立主循环):
  - `:1401 StartChildProcess(B_CHECKPOINTER)` → `postmaster/checkpointer.c:206 CheckpointerMain`(周期 checkpoint 刷脏页)。
  - `:1403 B_BG_WRITER` → `postmaster/bgwriter.c:89 BackgroundWriterMain`(后台预刷脏页,平滑 IO)。
  - `:3364 B_WAL_WRITER` → `postmaster/walwriter.c:89 WalWriterMain`(异步刷 WAL buffer)。
  - `:1408 B_STARTUP` → `postmaster/startup.c:216 StartupProcessMain`(崩溃恢复/回放 WAL)。
  - `:3375 B_AUTOVAC_LAUNCHER` → `postmaster/autovacuum.c:411 AutoVacLauncherMain`;worker `:1418 AutoVacWorkerMain`。
  - `:3422 B_WAL_RECEIVER` → `replication/walreceiver.c:154 WalReceiverMain`(备库收 WAL)。
- 不变量:后端崩溃被 postmaster 隔离(进程边界),必要时全体重启做崩溃恢复 → 稳健性 vs fork 开销的取舍。

## 主线 2 · 查询生命周期(parse→analyze→rewrite→plan→execute · Hot Path)
- `tcop/postgres.c:4364 PostgresMain` —— 后端主循环(读客户端消息,分派)。
- `:1030 exec_simple_query`(simple query 协议主体),阶段串联:
  - **parse**:`:1083 pg_parse_query` → `parser/parser.c:42 raw_parser`(裸语法树 RawStmt)。
  - **analyze+rewrite**:`:1213 pg_analyze_and_rewrite_fixedparams` → `parser/analyze.c:272 transformTopLevelStmt` / `:335 transformStmt`(语义分析成 Query),规则重写 `rewrite/rewriteHandler.c:4789 QueryRewrite`。
  - **plan**:`:1216 pg_plan_queries` → `optimizer/plan/planner.c:346 standard_planner`(Query → PlannedStmt)。
  - **execute**:`:1239 CreatePortal` → `:1258 PortalStart` → `:1297 PortalRun` → `:1306 PortalDrop`。
- 顺序不变量:一条 SQL 必经 5 阶段;多语句共享一个事务命令块(`:1064 start_xact_command` / `:1372 finish_xact_command`)。

## 主线 3 · 执行器(Volcano 拉取模型 · Hot Path)
- `executor/execMain.c:124 ExecutorStart` → `:143 standard_ExecutorStart`(建执行状态树)→ `:308 ExecutorRun` → `:318 standard_ExecutorRun`。
- `executor/execProcnode.c:142 ExecInitNode` 递归初始化计划节点树;`:448 ExecProcNodeFirst` → `:469 node->ExecProcNode(node)` 逐节点拉取。
- 拉取语义(`execProcnode.c:55-57` 注释):对顶节点反复调 `ExecProcNode()`,顶节点(如 `ExecNestLoop`)再对子节点调 `ExecProcNode()` → 火山模型,一次一元组自顶向下拉。
- 不变量:节点接口统一(Init/Proc/End),算子(SeqScan/IndexScan/NestLoop/HashJoin/Sort/Agg)可任意拼装。

## 主线 4 · 代价优化器(cost-based · Control Path)
- `optimizer/plan/planner.c:328 planner` → `:346 standard_planner` 入口。
- 关系路径生成:`optimizer/path/allpaths.c:183 make_one_rel`;连接搜索 `:3847 make_rel_from_joinlist`。
- 连接排序两条路:`:3952 standard_join_search`(动态规划,自底向上按 level 枚举)vs `:3916 geqo`(遗传算法 `optimizer/geqo/geqo_main.c:75 geqo`,表多时用,避免 DP 爆炸)。
- 代价估算 `optimizer/path/costsize.c`:`:271 cost_seqscan`、`:546 cost_index`、`:2202 cost_sort`、`:3396 initial_cost_nestloop` —— 用统计信息(行数/选择率)算 startup+total cost,选最便宜 path。
- 不变量:枚举等价 path,按估算代价取最小 → 声明式 SQL 到物理计划的核心决策点。

## 主线 5 · 存储：页/元组/缓冲池(State Path)
- 缓冲池:`storage/buffer/bufmgr.c:879 ReadBuffer` → `:926 ReadBufferExtended` → `:1276 ReadBuffer_common`(命中共享缓冲返回,未命中读盘)。
- 淘汰:`storage/buffer/freelist.c:184 StrategyGetBuffer` + `:104 ClockSweepTick` —— **时钟扫描(clock-sweep)** 近似 LRU 选牺牲页。
- 页布局:`storage/page/bufpage.c:42 PageInit`(pd_lower/pd_upper 划分)、`:203 PageAddItemExtended`(行指针数组自前向后、元组自后向前生长)。
- 堆元组读写:`access/heap/heapam.c:2004 heap_insert`、`:3201 heap_update`、`:2717 heap_delete`、`:1684 heap_fetch`。
- 不变量:所有数据页访问必过缓冲池(共享内存),脏页由 checkpointer/bgwriter 回刷。

## 主线 6 · MVCC 多版本并发(State Path · 核心张力)
- 可见性:`access/heap/heapam_visibility.c:939 HeapTupleSatisfiesMVCC` —— 按快照判断元组对当前事务是否可见(比对 xmin/xmax vs snapshot)。
- 版本标记(写时):`access/heap/heapam.c:2219 HeapTupleHeaderSetXmin`(插入者事务号)、`:2224 SetXmax(0)`;删除 `:3011 HeapTupleHeaderSetXmax`(不物理删,置 xmax)。
- 快照:`storage/ipc/procarray.c:2114 GetSnapshotData`(扫 ProcArray 取活跃事务的 xmin/xmax/xip)、`utils/time/snapmgr.c:272 GetTransactionSnapshot`。
- HOT 更新:`access/heap/heapam.c:3981 use_hot_update=true`(`:4029/:4159` 应用)—— 新版本同页且不改索引列时,不新增索引项,减膨胀。
- **张力兑现**:UPDATE/DELETE 只置 xmax 留旧版本 → 读走快照不加锁(读不阻塞写),但死元组累积成 bloat,须主线 11 回收。

## 主线 7 · WAL 预写日志(durability · Failure Path)
- 写 WAL:`access/transam/xloginsert.c:482 XLogInsert`(`:246 XLogRegisterBuffer`、`:372 XLogRegisterData` 组装 rdata)→ `access/transam/xlog.c:784 XLogInsertRecord`(入 WAL buffer,返回 LSN)。
- **WAL 规则铁证**:`xlog.c:779-781` 注释 "write the log before the data" —— LSN 是数据页刷盘前 WAL 必须先 flush 到的位点。
- 落地顺序:`heapam.c:2087 MarkBufferDirty` 改数据页 → 组 WAL → `:2164 PageSetLSN(page, recptr)` 打 LSN;刷脏页时 `bufmgr.c:4565 BufferGetLSN` → `:4585 XLogFlush(recptr)` 强制 WAL 先落盘。
- 刷盘/检查点:`xlog.c:2800 XLogFlush`(fsync 到 recptr)、`:7400 CreateCheckPoint`(建一致点,截断 WAL)。
- 恢复/复制:崩溃回放 `xlog.c:5845 StartupXLOG`;主库 `replication/walsender.c:3036 WalSndLoop`/`:850 StartReplication` 发流,备库 `walreceiver.c:154 WalReceiverMain` 收。
- 不变量:先日志后数据 → 崩溃后凭 WAL 回放到一致点。**张力**:commit fsync 保证持久但成写吞吐瓶颈。

## 主线 8 · 事务与提交(commit · clog · Failure Path)
- `access/transam/xact.c:2106 StartTransaction` → `:2270 CommitTransaction` → `:1345 RecordTransactionCommit`(写 commit WAL 记录)。
- **持久点**:`xact.c:1544 XLogFlush(XactLastRecEnd)`,条件 `:1540-1542`(synchronous_commit > OFF 或强制)—— commit 记录 fsync 后才算真正提交。
- 事务状态:`access/transam/clog.c:192 TransactionIdSetTreeStatus` —— 在 CLOG(commit log)标 xid 为 committed/aborted,MVCC 可见性据此判断。
- 不变量:commit 的原子锚点是 WAL 记录落盘 + CLOG 置位;此后该 xid 的 xmin/xmax 对后续快照生效。

## 主线 9 · 锁与死锁(locking · Failure Path)
- 常规锁管理器:`storage/lmgr/lock.c:806 LockAcquire` → `:833 LockAcquireExtended`;释放 `:2110 LockRelease`;冲突时 `:1229 WaitOnLock`。
- 等待/入睡:`storage/lmgr/proc.c:1378 ProcSleep`(进等待队列);清理 `:828 LockErrorCleanup`。
- 死锁检测:`proc.c:62 DeadlockTimeout=1000`(ms)超时触发 → `:1431 timeouts[0].delay_ms=DeadlockTimeout` → `storage/lmgr/deadlock.c:220 DeadLockCheck` → `:312 DeadLockCheckRecurse`(等待图找环)。
- 不变量:锁按表级/行级多种模式(ACCESS SHARE..ACCESS EXCLUSIVE)共存矩阵冲突判定;死锁不预防、超时后检测破环。

## 主线 10 · 索引访问方法(index AM · Hot/State Path)
- 通用 AM 接口:`access/index/indexam.c:134 index_open`、`:214 index_insert`、`:257 index_beginscan`、`:599 index_getnext_tid` → `:698 index_getnext_slot`。
- B-tree 实现(经 handler 注册回调):`access/nbtree/nbtree.c:118 bthandler`、`:206 btinsert`、`:230 btgettuple`、`:339 btbeginscan`。
- 不变量:AM 层抽象(btree/hash/gist/gin/brin 各实现同一组回调),索引返回 TID,再回堆取元组走 MVCC 可见性判断。

## 主线 11 · VACUUM / autovacuum(膨胀回收 · Optimization/State Path)
- 手动:`commands/vacuum.c:163 ExecVacuum` → `:494 vacuum` → `:2012 vacuum_rel` → 堆清理 `access/heap/vacuumlazy.c:624 heap_vacuum_rel`(扫堆标 LP_DEAD、清索引、回收行指针)。
- 自动:`postmaster/autovacuum.c:411 AutoVacLauncherMain`(按阈值调度)→ worker `:1418 AutoVacWorkerMain` → `:1928 do_autovacuum`。
- 不变量:回收主线 6 留下的死元组(xmax 已提交且对所有快照不可见者),推进 relfrozenxid 防 xid 回卷。
- **张力闭环**:这是 MVCC "读不阻塞写"所欠的债 —— 不清则 bloat 膨胀、性能退化。

## 主线 12 · 共享内存(shared memory · 协调基座)
- `storage/ipc/ipci.c:120 CreateSharedMemoryAndSemaphores` —— 一次性建全体共享内存;`:57 CalculateShmemSize` 估算大小,`:135 PGSharedMemoryCreate` 分配。
- 驻留其中的核心结构:共享缓冲池(主线 5)、WAL buffer(主线 7)、锁表(主线 9)、ProcArray(主线 6 快照来源)。
- 进程注册:`storage/lmgr/proc.c:393 InitProcess` / `:588 InitProcessPhase2`(每后端建 PGPROC 挂入),`storage/ipc/procarray.c:464 ProcArrayAdd`(加入活跃事务数组)。
- 不变量:进程模型下,跨后端的可见性/锁/缓冲一致性全靠这块 fork 前建好、fork 后继承的共享内存协调。

## 五路径归纳(供绘图 · 总架构图重画依据)
- **Hot Path**:client → backend `exec_simple_query` → parse→analyze/rewrite→`standard_planner`→`PortalRun`→`ExecutorRun`→`ExecProcNode` 拉取 → `ReadBuffer` 命中共享缓冲 → 堆/索引取元组 → MVCC 可见性过滤。
- **Control Path**:postmaster fork 后端与后台进程;优化器 `standard_planner`(DP/GEQO 选 path);事务 begin/commit 命令块。
- **State Path**:共享缓冲池 + 堆页/元组(xmin/xmax 多版本)+ 索引 AM(TID)+ CLOG 事务状态;快照 `GetSnapshotData` 从 ProcArray 派生。
- **Failure Path**:WAL 先写后刷("write log before data"),commit `XLogFlush` fsync;崩溃 `StartupXLOG` 回放;锁冲突 `WaitOnLock` + `DeadLockCheck` 超时破环;主备 walsender/walreceiver 流复制。
- **Optimization**:MVCC 无锁读、HOT 更新减索引膨胀、clock-sweep 缓冲淘汰、bgwriter/walwriter 异步刷、代价优化器选最便宜计划、autovacuum 回收 bloat。

## 总架构图重画蓝图(五路径工业剖面)

- 左 Control 栏:postmaster/ServerLoop、后台进程群(checkpointer/bgwriter/walwriter/autovacuum-launcher/startup)、代价优化器(DP/GEQO)。
- 中 Hot:backend 管线 → ExecProcNode 拉取 → ReadBuffer → 堆/索引访问。
- 右 State:共享缓冲池 + 堆页(MVCC 多版本 xmin/xmax)+ 索引 AM + CLOG;ProcArray 供快照。
- 底 Failure/Durability:WAL(write-log-before-data → XLogFlush fsync → CreateCheckPoint),崩溃恢复 StartupXLOG,主备 walsender→walreceiver 流复制,锁表 + 死锁检测。
- 贯穿张力标注:MVCC 读不阻塞写(左上读路)↔ 死元组膨胀 ← autovacuum 回收(底部回收箭头);WAL fsync 持久 ↔ 写吞吐;fork 隔离 ↔ 连接开销(外接连接池)。
