# raft 源码核实事实台账（source-verified fact ledger）

> 源码：`~/workdir/raft`(hashicorp/raft)。每条事实 = 从真实源码 grep/read 核实，标 `file:line`。
> 判型:**共识算法库**(Consensus Library) —— 不是完整服务,是嵌入宿主应用的复制状态机引擎。
> 接触面 = 编程 API(`Apply`/`AddVoter`/`Snapshot`/`VerifyLeader`);宿主提供 FSM/LogStore/StableStore/SnapshotStore/Transport 五大插件。
> 设计张力:**强一致(线性化)vs 可用性/吞吐** —— 多数派提交保证正确,pipeline/batch 缓解延迟。

## 判型依据(五大插件接口 = 库边界)
- `fsm.go:FSM` interface —— 宿主状态机(Apply/Snapshot/Restore)。
- `log.go:LogStore` interface —— 日志持久化。
- `stable.go:StableStore` interface —— 持久元数据(term/vote)。
- `snapshot.go:SnapshotStore` interface —— 快照存储。
- `transport.go:Transport` interface —— 节点间 RPC。
- `api.go:type Raft struct` —— 引擎主体。

## 主线 1 · 状态机与主循环(replicated state machine core)
- `state.go:13 type RaftState uint32`;`:17-23` 三态 `Follower`(iota=0) / `Candidate` / `Leader`。
- `raft.go:136 func (r *Raft) run()` 主循环;`:149/151/153` 按 state 分派 `runFollower()/runCandidate()/runLeader()`。
- `raft.go:159 runFollower`、`:286 runCandidate`、`:467 runLeader` 三态各自主循环。
- 不变量:任一 term 至多一个 Leader(靠多数派投票 + 每 term 每节点至多投一票,见主线2)。

## 主线 2 · Leader 选举(leader election)
- `raft.go:287 term := r.getCurrentTerm() + 1` —— 进 candidate 先自增 term。
- `raft.go:301(附近) voteCh = r.electSelf()` 发起投票;`votesNeeded := r.quorumSize()` 需简单多数。
- 支持 **pre-vote**(`raft.go:runCandidate` 内 `preVoteGrantedVotes` 计票)—— 防分区节点无谓抬高 term。
- 持久投票记录:`raft.go:28-30` `keyCurrentTerm/keyLastVoteTerm/keyLastVoteCand`;`:2154-2166 persistVote/setCurrentTerm` 写 StableStore(重启不重复投票,保证每 term 至多一票)。

## 主线 3 · 日志复制(log replication · Hot Path 主轨)
- `raft.go:1458 appendEntries(rpc, a *AppendEntriesRequest)` —— follower 侧 RPC 处理:一致性检查(prevLogIndex/Term)→ 截断冲突 → 追加 → 更新 commitIndex。
- Leader 侧:`raft.go:613 r.goFunc(func(){ r.replicate(s) })` 每 follower 一个复制 goroutine;`replication.go:137 replicate`、`:202 replicateTo`。
- 日志类型:`log.go:17-43` `LogCommand`(用户FSM) / `LogNoop`(断言领导权) / `LogBarrier`(屏障) / `LogConfiguration`(成员变更)。

## 主线 4 · 提交与应用(commit index · quorum)
- `commitment.go:14 type commitment` 追踪各 follower `matchIndexes`。
- `commitment.go:88-104 recalculate()`:排序 matchIndex,取 `matched[(len(matched)-1)/2]` = **多数派匹配点**,大于旧 commitIndex 且 ≥ startIndex(本 term)则推进并 `asyncNotifyCh(commitCh)`。
- 不变量:只提交**当前 term** 的日志条目(`startIndex` 约束),Raft 论文 §5.4.2 的安全性要点。

## 主线 5 · 持久状态(persistent state)
- `raft.go:28-30` StableStore 三键 `CurrentTerm`/`LastVoteTerm`/`LastVoteCand`;`:1709-1714 restoreState`、`:2154-2166` 落盘。
- 崩溃恢复不变量:term 单调不减、每 term 至多一票 —— 靠这三键持久化保证。

## 主线 6 · 快照与日志压缩(snapshot & log compaction)
- `snapshot.go:73 runSnapshots()` 后台循环;`:76 randomTimeout(SnapshotInterval)`;`:107 shouldSnapshot()` 按 `TrailingLogs` 判定。
- Leader 追不上落后 follower 时发快照:`replication.go:299 sendLatestSnapshot`。
- follower 侧:`raft.go:1837 installSnapshot`;`:1969 compactLogs` 截断已快照日志。

## 主线 7 · 成员变更(membership change · single-server)
- `raft.go:1201 appendConfigurationEntry` —— 成员变更走一条 `LogConfiguration` 日志(而非单独协议)。
- `raft.go:116 requestConfigChange` / `AddVoter`/`RemoveServer`(api.go)。
- 采用 **single-server change**(一次加/减一个节点),避免联合共识复杂度 —— 每次配置差异 ≤1 保证新旧多数派必相交。

## 主线 8 · 传输与流水线(transport · pipeline · Opt Path)
- `replication.go:388 heartbeat` 独立 goroutine 发空 AppendEntries 维持领导权(与日志复制解耦)。
- `replication.go:446 pipelineReplicate` —— **流水线复制**:不等前一个 AppendEntries 响应就发下一个,提升吞吐(Opt Path)。
- `net_transport.go` / `tcp_transport.go` 具体传输;`inmem_transport.go` 测试用。

## 主线 9 · 领导权验证与转移(read-index · leadership transfer)
- `raft.go:VerifyLeader`(api.go)—— 线性化读:确认自己仍是 Leader(向多数派确认)才响应读,避免脑裂旧 Leader 返回陈旧值。
- `raft.go:91 leadershipTransferInProgress`、`:443 setLeadershipTransferInProgress`、`:692 leadershipTransferCh` —— 主动领导权转移(优雅交接)。

## 五路径归纳(供绘图)
- **Hot Path**:client `Apply` → Leader 追加日志 → `replicate` 并行发 follower → 多数派 `matchIndex` → `commitment.recalculate` 提交 → FSM.Apply。
- **Control Path**:选举(term++/electSelf/pre-vote)、heartbeat 维持领导权、成员变更(LogConfiguration)。
- **State Path**:StableStore(term/vote 持久)+ LogStore(日志)+ SnapshotStore(快照)+ FSM(应用态)。
- **Failure Path**:网络分区 → 心跳超时 → 重新选举;落后 follower → installSnapshot 追赶;旧 Leader → VerifyLeader 拒绝陈旧读;脑裂 → 多数派约束 + pre-vote 防治。
- **Optimization**:pipelineReplicate(流水线)、log batching、heartbeat 与复制解耦、TrailingLogs 保留窗口。
