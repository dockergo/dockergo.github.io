# etcd 源码核实事实台账（source-verified fact ledger）

> 源码：`~/workdir/etcd`。每条事实从真实源码 grep/read 核实,标 `file:line`。
> 判型:**分布式 KV 控制面**(distributed control-plane store) —— Raft 复制 + MVCC 多版本 + Watch 事件 + Lease 租约,统一在一个控制面状态机上。
> 现状问题:主图散成 Raft/KV/Watch/Lease 局部图,缺统一"控制面状态核心"叙事。
> 设计张力:**线性一致(强)vs 读吞吐** —— 写走 Raft 多数派,读用 ReadIndex 免走日志但仍保证线性化。

## 主线 1 · 写路径(propose → raft → apply · Hot Path)
- `server/etcdserver/v3_server.go:295 Put` / `:325 Txn` —— 写入口。
- `:1012 raftRequest` → `:1058 processInternalRaftRequestOnce` —— 序列化成 `InternalRaftRequest` 提交给 raft,等 applied 通知。
- 应用侧:`server/etcdserver/server.go:1881 apply(...)` 遍历已提交 entry;`:1934 applyEntryNormal` 把普通 entry 交 applierV3 落到 MVCC。
- `server/etcdserver/raft.go:404 apply() chan toApply` —— raftNode 把已提交条目送出供状态机应用。
- 不变量:所有写经 Raft 多数派复制并按 index 顺序 apply,各节点状态机确定性收敛。

## 主线 2 · 线性一致读(ReadIndex · Control Path)
- `server/etcdserver/read/read.go:96 LinearizableReadLoop` —— 独立循环处理线性读请求。
- `:148 requestCurrentIndex` → `:152 sendReadIndex` —— 向 raft 请 ReadIndex:确认当前 leader + 拿到 commit index。
- `:74 LinearizableReadNotify` —— 读请求等待本地 applied index ≥ ReadIndex 才返回,保证读到"发起时刻已提交"的最新值。
- 机制要点:**读不写日志**(不 propose),只确认 leader 租约 + 等 applied 追上 → 线性一致 + 高读吞吐。这是 etcd 读性能关键。

## 主线 3 · MVCC 多版本存储(State Path)
- `server/storage/mvcc/index.go:39 treeIndex` —— 内存 B 树(`btree.BTree[*keyIndex]`,`:41`),key → `keyIndex`(各版本 revision 链)。
- 数据落 boltdb backend,key = revision(main.sub),value = protobuf KV。treeIndex 只存 key→revision 映射,值查 backend。
- 版本语义:每次写 global revision++;读按 revision 查历史版本 → Watch/一致性快照/compaction 基础。
- compaction 回收旧版本(keyIndex 裁剪 + backend 删除)。

## 主线 4 · Watch 事件(State→通知)
- `server/storage/mvcc/watchable_store.go:56 watchableStore` 包裹 mvcc store。
- `:224 syncWatchersLoop` 后台循环;`:346 syncWatchers` —— 把落后的 watcher(unsynced)从 backend 按 revision 追赶到当前,推送事件。
- 分 synced / unsynced 两组:实时写直接通知 synced;新建或落后的进 unsynced 由 syncWatchers 追赶 → 背压隔离。

## 主线 5 · Lease 租约(Control · TTL)
- `server/lease/lessor.go:83 Lessor` interface;`:281 Grant(id, ttl)` 建租约;`:326 Revoke` 撤销(级联删除绑定 key)。
- checkpoint 机制:leader 定期把租约剩余 TTL 作为 checkpoint 走 raft 复制,防 leader 切换后 TTL 重置(follower 提升为 leader 仍知剩余时间)。
- 过期:leader 检测过期租约 → 发 Revoke 走 raft → apply 删除绑定 key。

## 主线 6 · 一致性与故障(Failure Path)
- leader 切换:`leaderChangedNotifier`(read.go:117)—— leader 变更时未决线性读失败重试,防陈旧。
- 网络分区:少数派侧无法达成多数派 → 写阻塞、线性读失败(CP 取向)。
- 快照:raft 日志过大触发 snapshot,落后 follower 收 InstallSnapshot 追赶。

## 五路径归纳(供绘图 · 总架构图重画依据)
- **Hot Path**:client Put/Txn → processInternalRaftRequestOnce → raft 多数派 → apply → applyEntryNormal → MVCC treeIndex+backend。
- **Control Path**:线性读 ReadIndex loop(read.go:96/148)、Lease Grant/checkpoint、成员变更。
- **State Path**:MVCC treeIndex(内存 btree)+ boltdb backend(revision KV);Watch 从 state 派生事件。
- **Failure Path**:leader 切换→线性读重试;分区→少数派拒写(CP);日志过大→snapshot 追赶。
- **Optimization**:ReadIndex 免写日志读、Watch synced/unsynced 分组背压、Lease checkpoint、batch apply。
