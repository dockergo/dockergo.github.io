## 一、判型（系统类型一句话）

**高吞吐 LLM 推理引擎 · PagedAttention 分块 KV 缓存 + 连续批处理 + EngineCore step 循环。**

- 不是训练框架、不是通用服务框架：它把"训练好的大模型"包成一个**每步(step)取一批请求、跑一遍前向、采一个 token、循环到出完**的 token 生成服务。
- 灵魂 = 把 KV 缓存像 OS 分页一样切成**固定大小物理块**（默认 16 token/块）消除显存碎片，让连续批处理把 GPU 填满。
- 核心真源：`vllm/v1/engine/core.py:EngineCore.step` — `schedule() → execute_model() → sample_tokens() → update_from_output()`。

---

## 二、设计张力（显式 X vs Y）

**显存利用率 / 吞吐（PagedAttention 分块消碎片 + 连续批处理填满 GPU）  vs  调度与间接开销（块表间接寻址、抢占即重算丢弃已算 KV）。**

- 得（吞吐侧）：`KVCacheManager.allocate_slots` 按需一块块分配（`kv_cache_manager.py:283`）；`BlockPool` 用 `free_block_queue` + `ref_cnt` + `cached_block_hash_to_block` 做分配/回收/前缀共享（`block_pool.py:143`）；`Scheduler.schedule` 每步按 token 预算重组批（`scheduler.py:417`）。
- 失（复杂度侧）：读 KV 要过**块表**间接寻址（逻辑块→物理块，`gpu_model_runner.py` block_table + slot_mapping）；显存不足时**抢占 = 重算**——`_preempt_request` 直接 `_free_request_blocks` + `num_computed_tokens = 0` 把请求打回 waiting 队列重跑（`scheduler.py:1173-1195`）。
- 关键校正：**v1 抢占是 recompute-only**。`RequestStatus` 无 `SWAPPED`（`request.py:346-356` 只有 WAITING/RUNNING/PREEMPTED/FINISHED_*）；`vllm/v1/core/` 下 grep 无 `swap_out/swap_in`。故台账写"重算式抢占"，**不写换出/swap**（那是 v0 语义）。

## 三、五路径归纳

### Hot（推理 step 出 token）— 主脊
- `EngineCore.step` (`core.py:546`)：`schedule()`→`execute_model(non_block=True)`→`future.result()`→`sample_tokens()`→`update_from_output()`。
- 前向：`GPUModelRunner.execute_model` (`gpu_model_runner.py:4097`) 跑一遍模型，`_prepare_inputs` (`:1930`) 建 attn_metadata + block_table + slot_mapping。
- 采样：`GPUModelRunner.sample_tokens` (`:4483`) → `_sample` (`:3623`)；`Sampler.forward` (`vllm/v1/sample/sampler.py:72`) logits→token（greedy=argmax / random，temperature/top_p/top_k）。
- 回填/判停：`Scheduler.update_from_output` (`scheduler.py:1533`) append 新 token、`check_stop`；未停 → 下一 step 重进 running。
- 入口：离线 `LLM.generate` (`vllm/entrypoints/llm.py:411`)；在线 `vllm/entrypoints/openai/api_server.py` 经 `EngineClient`。

### Control（连续批处理调度）
- `Scheduler.schedule(throttle_prefills)` (`scheduler.py:417`)：`token_budget = max_num_scheduled_tokens` (`:110,437`)，先扫 `self.running` 再收 `self.waiting`（`:463,661`）。
- 连续批处理：随到随入（waiting 队列）、完成即出（stopped 移除），**每步重组批**，非静态 batch。
- chunked prefill / long-prefill 节流：`is_prefill_chunk`、`long_prefill_token_threshold`、`throttle_prefills`（`:459,488,499`）。

### State（分块 KV 缓存 · 中心资源）
- 块大小默认 16：`CacheConfig.DEFAULT_BLOCK_SIZE = 16` (`vllm/config/cache.py:47,255`)。
- `KVCacheManager` (`kv_cache_manager.py:114`)：`allocate_slots` (`:283`) 按需给块；`get_computed_blocks` (`:207`) 查前缀命中；委托 `KVCacheCoordinator`（`kv_cache_coordinator.py`）。
- `BlockPool` (`block_pool.py:143`)：`get_new_blocks` (`:647`) 从 `free_block_queue` popleft，`ref_cnt` 引用计数共享，`cache_full_blocks` (`:225`) 按 hash 登记，`_maybe_evict_cached_block` (`:679`) LRU 式驱逐。
- 前缀缓存复用：`find_longest_cache_hit`（`kv_cache_coordinator.py:369` 等）—— 公共前缀块命中即复用，只算一次。
- PagedAttention 写读：`PagedAttention.write_to_paged_cache` → `ops.reshape_and_cache`（`vllm/v1/attention/ops/paged_attn.py:32`）按 slot_mapping 写块；后端 `vllm/v1/attention/backends/flash_attn.py` 等按 block_table gather。CUDA 核在 `csrc/`（`csrc/rocm/attention.cu`，`csrc/libtorch_stable/cache_kernels.cu`）。

### 失败（抢占 → 重算）
- `_preempt_request` (`scheduler.py:1173`)：`_free_request_blocks` 释放物理块 → `status = PREEMPTED` → `num_computed_tokens = 0` → `waiting.prepend_request` 打回队首重跑。
- 触发点：`allocate_slots` 返回 None（无空闲块）→ 循环里 `self.running.pop()` 抢最低优先级（默认 FCFS 抢队尾），`scheduler.py:553-603`。
- **重算式，非换出**：丢弃已算 KV、下次重新 prefill；v1 无 swap 路径（见张力节校正）。

### 横切（分布式并行）
- `initialize_model_parallel(tp_size, pp_size)` (`vllm/distributed/parallel_state.py:1713`)，建 `_TP` (`:1805`) / `_PP` (`:1868`) group。
- TP 层内切：`ColumnParallelLinear` (`linear.py:397`) / `RowParallelLinear` (`linear.py:1537`) / `VocabParallelEmbedding`；forward 里 all-reduce / all-gather（`parallel_state.py:130,641`）。
- PP 层间切：按层段分卡传激活。
- 多进程执行器：`MultiprocExecutor` (`vllm/v1/executor/multiproc_executor.py:103`) 起 `WorkerProc` (`gpu_worker.py`)，`rpc_broadcast_mq` 广播 scheduler_output。各卡各存自己那份 KV 分片。
- 显存探测/建块：`GPUWorker.determine_available_memory` (`gpu_worker.py:448`) → `num_gpu_blocks` (`:720`)。
