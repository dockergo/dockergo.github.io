## 判型（这是哪一类系统）

**分布式 task/actor 运行时**：单一应用门户 CoreWorker（driver 与 worker 内嵌同一套核心，`core_worker/core_worker.h:182 class CoreWorker`，`GetWorkerType()` 区分 DRIVER/WORKER），向下分三条正交机制：

1. **去中心化调度（owner 直接派活 / worker 直连租借）** — 提交方 CoreWorker 先向 raylet 租 worker，租到后**直接把 task 推给被租 worker**，不经中央调度器中转：
   - `core_worker/task_submission/normal_task_submitter.cc:333 RequestWorkerLeaseRequest` + `:339 RequestWorkerLease`（向 raylet 请求租约）
   - 同文件 `:185 PushNormalTask(addr, client, ...)`（租到 worker 后直推，addr = 被租 worker 地址）
2. **GCS 全局控制存储** — head 节点单进程 GCS，集中放集群级元数据（节点表 / actor 注册 / KV / pub-sub）：
   - `gcs/gcs_server.cc:316-341`（InitGcsNodeManager / InitGcsResourceManager / InitGcsPlacementGroupManager / InitGcsActorManager / InitGcsAutoscalerStateManager 逐一在同一 GcsServer 内装配）
3. **Plasma 共享内存对象存储** — 每节点一个 plasma store，大对象走共享内存零拷贝（mmap）：
   - `object_manager/plasma/store.cc:174 PlasmaStore::CreateObject`，`:206 mmap_sizes`（创建对象即分配 mmap 段，客户端映射同一物理内存）

一句话判型：
**分布式 task/actor 运行时 · 去中心化调度(worker 直连租借) + GCS 全局控制存储 + Plasma 共享内存对象存储**

## 设计张力（本系统的定义性矛盾）

**去中心化低延迟 vs 全局状态一致**

- 低延迟一侧：细粒度对象元数据**不进 GCS**，由**对象 owner**（创建该 ObjectRef 的 worker）自己记引用计数、自己追踪对象位置。
  - `core_worker/reference_counter.h:42-44`：ReferenceCounter「Class used by the core worker to keep track of ObjectID reference counts」——引用计数在 worker 本地，非全局。
  - `core_worker/task_manager.cc:374 reference_counter_.AddOwnedObject(return_id, ...)`：task 提交方在本地把返回对象登记为「自己拥有」，owner = 调用者。
  - `object_manager/ownership_object_directory.cc:35 class OwnershipBasedObjectDirectory`：对象**位置目录按 owner 维护**，而非 GCS 集中目录。
  - task 直投：`normal_task_submitter.cc:185 PushNormalTask` 绕开中央调度器。
- 一致性一侧：集群级、粗粒度、需全局共识的状态**集中放 GCS**——节点成员表、actor 注册表、KV、pub/sub 广播，单点真源。
  - `gcs/gcs_node_manager.h:47 class GcsNodeManager`（节点表 + `:122 OnNodeFailure` 故障判定）
  - `gcs/actor/gcs_actor_manager.h`（`registered_actors_` / `created_actors_` 注册表，见 61-82 行状态机注释）
  - `gcs/gcs_kv_manager.h:103 class GcsInternalKVManager`（KV）
  - `pubsub/gcs_publisher.h:49 class GcsPublisher`（`:72 PublishActor` / `:76 PublishNodeInfo` 广播）

矛盾点：owner 直连派活/引用不进 GCS → 低延迟、GCS 不成瓶颈；但对象丢失/owner 死亡需靠 **lineage 重算**兜底，而集群成员与 actor 归属又必须 GCS 集中裁决。两者在同一系统内共存 = Ray 的定义性张力。

张力铭牌文案：
**去中心化低延迟(owner 直接派活、对象引用不进 GCS) vs 全局状态一致(GCS 集中裁决节点/actor 元数据)**
