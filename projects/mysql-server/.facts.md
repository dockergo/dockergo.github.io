## 一、判型

**可插拔存储引擎的关系数据库** · SQL 层（解析 / 优化 / 执行）与存储引擎经 **handler API** 解耦；默认引擎 **InnoDB** = B+树聚簇索引 + Buffer Pool + redo/undo 的 MVCC 事务引擎。

判型支撑锚点：
- SQL 层引擎无关：`sql/sql_parse.cc:dispatch_command@1218`、`sql/sql_optimizer.cc:JOIN::optimize@138`、`sql/sql_executor.cc:JOIN::exec@104`
- 可插拔缝（MySQL 的定义性抽象）：`sql/handler.h:class handler@2157`（每表实例）、`sql/handler.h:struct handlerton@698`（每引擎单例）、`sql/handler.h:(*create)(handlerton*,TABLE_SHARE*,MEM_ROOT*)@772`
- 默认引擎 InnoDB：`storage/innobase/handler/ha_innodb.cc:innobase_commit@971`、`storage/innobase/row/row0sel.cc:row_search_mvcc@4674`、`storage/innobase/btr/btr0cur.cc:btr_cur_search_to_nth_level@745`

## 二、设计张力

**主张力（灵魂）：SQL 层引擎无关（handler API 可插拔） ⟷ InnoDB 事务/崩溃恢复深度耦合。**
解耦切出一条 handler API 缝：Server 层不懂行如何存储，只调虚函数（`ha_write_row` / `index_read_map` / `ha_external_lock`）。代价 = **双日志问题**：Server 层记逻辑 **binlog**（复制/时点恢复用），引擎层记物理 **redo**（崩溃恢复用），两份日志必须一致 → 靠 **2PC/XA + 组提交** 对账。
- 缝：`sql/handler.h:ha_write_row@2437`、`sql/handler.h:ha_external_lock@2436`、`sql/handler.h:index_read_map@2819`
- 双日志 2PC：`sql/handler.cc:ha_commit_trans@1671`（`tc_log->prepare@1792` → `tc_log->commit@1807`）、`sql/binlog.cc:binlog_prepare@1818`、`sql/binlog.cc:MYSQL_BIN_LOG::ordered_commit@9520`
- handlerton 事务钩子：`sql/handler.h:(*prepare)@768`、`(*commit)@766`、`(*rollback)@767`

**副张力：ACID 持久（redo WAL + double-write + fsync） ⟷ 写吞吐（组提交 / 异步刷脏）。**
- WAL：`storage/innobase/log/log0log.cc:log_write_up_to@1217`；mtr_commit 写 redo（`log0log.cc@72` 注释）
- double-write 防页断裂：`storage/innobase/buf/buf0dblwr.cc:buf_dblwr@51`
- 组提交三阶段：`sql/binlog.cc:Stage_manager::enroll_for@2123`（FLUSH/SYNC/COMMIT）、`process_commit_stage_queue@9013`、`sync_binlog_file@9224`
- 异步刷脏：`storage/innobase/buf/buf0flu.cc:buf_page_cleaner_is_active@84`（page cleaner 线程）

## 四、总架构图重画蓝图（house style · 分层脊）

viewBox `0 0 1080 660`（width=height=viewBox，过 linter）。自上而下分层，节点只挂机制概念名（无 file:line、无文件名、无左侧竖条）。

- **标题带 · 判型**（sharp）：可插拔存储引擎的关系数据库 —— SQL 层解耦 × InnoDB 深度耦合；副标 = 分层脊 + 源码基准 v5.7.44
- **设计张力铭牌带**（显式 nameplate）：handler API 可插拔 ⟷ InnoDB 事务耦合；派生双日志 2PC；ACID 持久 ⟷ 写吞吐
- **五型语义图例**：Hot 绿 / Control 蓝 / State 琥珀 / 失败 橙 / 横切 紫
- **powers 立身带**：世界最流行开源关系库；两层架构 + 可插拔引擎是其定义性设计
- **① 连接层**（hotspot→连接与线程模型）：client → mysqld 监听 → conn_handler（per-thread / one-thread）→ 每连接一个 **THD** 会话对象 → do_command 主循环
- **② SQL 层（Server 层，引擎无关）**：**Parser**（词法语法 → parse tree）→ **Optimizer**（基于代价选计划）→ **Executor**（volcano 嵌套循环迭代）；旁挂 **binlog**（Server 层逻辑日志，复制/时点恢复）。热区：SQL执行生命周期 / 查询优化器 / binlog与复制
- **③ handler API 抽象边界（可插拔缝 ★灵魂）**：**handlerton**（每引擎单例）+ **handler**（每表实例）虚函数集：write_row / index_read / external_lock / commit / prepare。热区：handler存储引擎API
- **④ InnoDB 存储引擎**（引擎层，深度耦合的事务/恢复）：
  - **Buffer Pool**（缓存页 + LRU + flush_list）· **B+树聚簇索引**（主键即数据）—— 热区：BufferPool与B+树
  - **redo log（WAL）** · **double-write buffer** —— 热区：redo日志与崩溃恢复
  - **undo log + ReadView（MVCC）** · **lock manager（行锁/死锁检测）** · **purge 线程** —— 热区：InnoDB事务与MVCC
- **⑤ 持久化底盘**：数据文件 .ibd（聚簇索引页）· redo log 文件 · doublewrite · binlog 文件
- **横切复制回路**：master binlog dump → slave relay log → SQL applier（紫）

读法：一条 SQL 从连接层进入 → SQL 层解析/优化/执行 → 穿过 handler API 缝调 InnoDB → 命中 Buffer Pool 或经 B+树取页 → 事务改动写 redo(WAL先行)+undo，提交时 binlog 与 redo 经 2PC 组提交对账 → 崩溃后 redo 重放 + double-write 修复 + undo 回滚。

- 全景主线框架 → powers 立身带
- 接口_SQL执行生命周期 → ② SQL 层标题区

- 支撑_查询优化器 → Optimizer 节点
- 支撑_handler存储引擎API → ③ 可插拔缝

- 支撑_InnoDB事务与MVCC → undo/ReadView/lock 节点
- 支撑_redo日志与崩溃恢复 → redo/double-write 节点
- 支撑_binlog与复制 → binlog 节点 + 横切回路
