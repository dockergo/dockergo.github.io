## 一、判型（一句定性）

**开放表格式 · 不可变快照 + 三层元数据(metadata.json → manifest list → manifest)的乐观并发表抽象 · 计算引擎无关。**

依据：
- **开放表格式，自身不读写数据/不执行查询**：`Table` 是给引擎链接的库 API —— `newScan()`（`api/.../Table.java:51`）、`currentSnapshot()`（:159）、`updateSchema()`（:188）、`updateSpec()`（:196）、`newAppend()`（:224）、`newRewrite()`（:247）。实现 `core/.../BaseTable.java`。真正读写 batch 由引擎（Spark/Trino/Flink）做。
- **根元数据是 JSON**：`core/.../TableMetadataParser.java:48,71`（文件名 `*.metadata.json`）；根字段见 `TableMetadata.java:245-294`（metadataFileLocation / schemas / specs / sortOrders / currentSnapshotId / snapshots / refs）。
- **三层不可变元数据树**：
  - metadata.json（根，多 schema/spec/snapshot 列表）
  - → `Snapshot.manifestListLocation()`（`api/.../Snapshot.java:171`）指向 **manifest list**
  - → manifest list 列出多个 **manifest file**（`ManifestFile`，`api/.../ManifestFile.java:198` partitions() 分区摘要）
  - → 每个 manifest 列出 **DataFile / DeleteFile 条目**（`api/.../DataFile.java`）
  - manifest list + manifest 默认 **Avro** 编码（`core/.../ManifestFiles.java:61` `MANIFEST_AVRO_SCHEMA`；v4 起支持 Parquet manifest，`TableMetadata.java:60` `MIN_FORMAT_VERSION_PARQUET_MANIFESTS=4`，`ManifestFiles.java:147` 注释 "non-Avro manifest formats"）。
- **快照不可变、每提交产新快照记父**：`Snapshot.snapshotId()`（`api/.../Snapshot.java:49`）、`parentId()`（:56）、`sequenceNumber()`（:42）。
- **乐观并发提交（OCC）**：见下节。

## 二、设计张力（显式铭牌）

**张力①（主）**：快照隔离 + 乐观并发（每次提交 = 一个新不可变快照，冲突则 refresh 重放重试）**vs** 三层元数据的读放大 / 小文件与元数据膨胀。
- 快照隔离/OCC：`core/.../SnapshotProducer.java:480-522` `commit()` —— `Tasks.foreach(ops).retry(COMMIT_NUM_RETRIES).exponentialBackoff(...).onlyRetryOn(CommitFailedException.class).run( taskOps -> { Snapshot s = apply(); ... taskOps.commit(base, updated.withUUID()); })`；`apply()`（:293-294）每次先 `refresh()` 重读 base。
- 读放大/膨胀的代偿：需后台 compaction 合并小文件 + expire snapshots 清历史（图中 ⑨ 表维护）。每次提交都新增 metadata/manifest-list/manifest 文件，读路径要逐层下钻。

**张力②**：计算引擎无关的开放格式抽象 **vs** 性能依赖与存储/计算的深度绑定。
- 引擎无关：`Table` 库 API（上引），Iceberg 只改元数据不碰数据。
- 绑定代偿：真实读写、向量化、谓词下推执行落在引擎侧；扫描规划产出 `FileScanTask` 交引擎读（`core/.../DataTableScan.java:64-91`）。

## 三、五路径归纳（时机 × 语义分型）

### 路径 A · Hot 读扫描（绿）
`core/.../DataTableScan.java:64-91` `doPlanFiles()`：取 `snapshot()` → `snapshot.dataManifests(io)` / `deleteManifests(io)` → 建 `ManifestGroup` → `planFiles()` 产 `FileScanTask` 交引擎读。
**两级剪枝**（`core/.../ManifestGroup.java`）：
- 一级（manifest 级）：`:279-309` `ManifestEvaluator.forPartitionFilter(...).eval(manifest)` 用分区摘要跳过整个 manifest（记 `scanMetrics.skippedDataManifests`）。
- 二级（文件级）：`:344-350` `ManifestFiles.read(manifest).filterRows(dataFilter).filterPartitions(partitionFilter)` 用每文件列统计（min/max/null）剪 DataFile。
文件列统计字段：`api/.../DataFile.java:57` VALUE_COUNTS、:63 NULL_VALUE_COUNTS、:75 LOWER_BOUNDS、:81 UPPER_BOUNDS。

### 路径 B · Hot 写提交（绿）
`core/.../SnapshotProducer.java:293-329` `apply()`：`refresh()` → 算 `sequenceNumber = base.nextSequenceNumber()`（:297）→ 生成新 manifest（`apply(base, parent)`）→ `ManifestLists.write(...)` 写新 manifest list（:306）→ 组装新 `Snapshot`。合并型写走 `core/.../MergingSnapshotProducer.java:69`（append/overwrite/delete 的抽象基类）。

### 路径 C · Control 快照乐观并发（蓝）
`core/.../SnapshotProducer.java:485-522`：retry 循环里 `taskOps.commit(base, updated)`；契约见 `api/.../TableOperations.java:49-51`「Implementations must check that the base metadata is current to avoid overwriting updates」= compare-and-swap。

### 路径 D · State 三层元数据 + 快照落地（琥珀）
metadata.json（`TableMetadataParser.java:71`）→ Snapshot（`Snapshot.java:171` manifestListLocation）→ manifest list（avro）→ manifest（`ManifestFiles.java:61`，`ManifestFile.java:198` 分区摘要）→ DataFile 条目（`DataFile.java`，content 0/1/2 见 `api/.../FileContent.java:25-27` DATA/POSITION_DELETES/EQUALITY_DELETES）。全部不可变文件。

### 路径 E · 失败：提交冲突重试（橙）
并发提交下 `TableOperations.commit(base, metadata)` 检出 base 非 current → 抛 `CommitFailedException`：
- Metastore 系：`core/.../BaseMetastoreTableOperations.java:109-119` `if (base != current()) throw new CommitFailedException("Cannot commit: stale table metadata")`。
- Hadoop 原子 rename 系：`core/.../hadoop/HadoopTableOperations.java:132-134` stale 检查 + `:161-162` `renameToFinal(...)` = 原子提交。
`SnapshotProducer.commit` 的 `.onlyRetryOn(CommitFailedException.class)`（:492）捕获后 `apply()` 重新 `refresh()` 重放，指数退避（:487-491）。

### 横切 · schema 演进 + 分区演进（紫）
- **schema 演进靠不可变字段 ID**：列用 field ID 追踪而非名字/位置；`api/.../UpdateSchema.java:60`「its field IDs are reassigned when added」；`api/.../Schema.java:267` highest field ID；rename/加删列/类型提升都不重写数据。
- **隐藏分区**：`api/.../PartitionSpec.java:50-51`「Partition data is produced by transforming columns ... represented by a named PartitionField」经 `Transform` 派生，用户不需在查询里写分区列。
- 每 manifest 记 `partitionSpecId`（`ManifestGroup.java:285` `specsById.get(specId)`），改分区规则不重写老数据。
- 贯穿 seq number：每提交递增（`SnapshotProducer.java:297`），快照/manifest 条目/行级删除都带 seq，删除靠 seq 比较决定作用范围。
