## 判型
**基于 HTTP/2 的高性能跨语言 RPC 框架**：Protobuf 生成强类型 Stub/Skeleton 契约 → `Channel`/`Server` 抽象承载一次调用 → 客户端 Channel 内是一条**可插拔装配线**（name-resolver 解析地址 → load-balancer 产出 picker 选连接 → filter 栈/interceptor 链过滤 → Subchannel 建连）→ 落到**客户端/服务端共用的 HTTP/2 transport**（多路复用 stream、帧化、流控、HPACK）→ 事件经**异步 CompletionQueue / EventEngine** 交付。控制面（resolver/LB）与数据面（transport）解耦，一切策略皆可替换。

## 设计张力（显式铭牌）
- **主张力**：跨语言统一 RPC 抽象 + 三处可插拔策略（resolver / load-balancer / filter 栈）—— vs —— HTTP/2 多路复用与流控带来的实现复杂度（单连接 N stream、帧交织、窗口对账）。
- **副张力**：异步高吞吐（CompletionQueue/EventEngine 非阻塞、单连接多路复用）—— vs —— 编程模型复杂度（tag 驱动、promise 化 CallSpine、Party 调度）。

## 五路径归纳（图例语义）
1. **Hot（绿，请求/响应流）**：Stub→Channel→CallSpine→(LB pick)→Subchannel→HTTP/2 transport→wire→(server) transport→filter 栈→方法 handler→响应回流。
2. **Control（蓝，解析+LB+分发）**：resolver 出地址→LB 消费地址产 picker→CallSpine 用 picker 选 subchannel；服务端 RequestMatcher `MatchOrQueue` 把 incoming call 派给等待的 RequestedCall。
3. **State（琥珀，HTTP/2 流 + 流控）**：单连接内 stream 生命周期、`TransportFlowControl`/`StreamFlowControl` 窗口对账、BDP 估计、subchannel 连接态。
4. **失败（橙，重试+超时）**：`retryPolicy`（maxAttempts + 退避 + RetryThrottle）重试；deadline 超时（GrpcTimeoutMetadata 编码）；keepalive ping 探活。
5. **横切（紫，凭证+拦截器）**：传输凭证（TLS/ALTS/local/insecure）握手 + 调用凭证注入 metadata；client/server interceptor 链、filter 栈横切每次调用。
