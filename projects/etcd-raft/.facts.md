## 判型：共识状态机核（host-driven · Ready 外包 I/O）

- **不是可嵌入的"电池全含"库，而是最小共识状态机内核。** 库只持内存中的协议状态机；无内置传输、无存储 I/O、无驱动复制的 goroutine。
- **宿主用 Node 循环驱动。** 主循环 `func (n *node) run()`（`node.go:343`）是 `for{ select{} }`：从 `tickc`/`propc`/`recvc`/`readyc`/`advancec` 等 channel 取事件，推进内部 `raft`。`StartNode`（`node.go:271`）里 `go n.run()` 起这个唯一 goroutine。
- **四动作契约**：`Node` 接口（`node.go:132-243`）定义 `Tick()`（:135）、`Step(ctx,msg)`（:156）、`Ready() <-chan Ready`（:164）、`Advance()`（:178）。`Propose`（:140）本质是 `Step` 一条 `MsgProp`（见 `node.go:471-472`）。
- **Ready = 一次 I/O 待办清单，全字段只读。** `type Ready struct`（`node.go:52-115`）：`Entries`（待持久化，注释 "saved to stable storage BEFORE Messages are sent" :74-80）、`Messages`（待发送 :98-110）、`CommittedEntries`（待应用 :89-96）、`Snapshot`（:82-87）、`HardState`（:58-66）、`ReadStates`（:68-72）。行内注释 "All fields in Ready are read-only"（:51）。
- **库自己声明 I/O 是宿主的责任。** `doc.go:69-86`："you must read from the Node.Ready() channel … 1. Write HardState, Entries, and Snapshot to persistent storage … 2. Send all Messages to the nodes named in the To field"。规范驱动循环样例见 `doc.go:123-145`（`saveToStorage / send / process / Advance`）。
- **Storage 是宿主实现、库只读的接口。** `type Storage interface`（`storage.go:48-96`）：`InitialState`（:56）、`Entries`（:77）、`Term`（:83）、`LastIndex`（:85）、`FirstIndex`（:90）、`Snapshot`（:95）。`MemoryStorage`（`storage.go:104`）只是参考实现，"may be implemented by the application"（:42-43）。

## 设计张力：正确性/最小内核  vs  I/O 全外包给宿主的复杂度

- **最小内核换来的：确定性与可测。** `raft` 结构（`raft.go:343-437`）纯内存态；`step stepFunc`（:426）与 `tick func()`（:425）是函数指针，无隐藏线程。`RawNode` 明言 "thread-unsafe Node"（`rawnode.go:31`），把并发也交给宿主。
- **代价：宿主要自己做 WAL / 传输 / 线程 / 重试。** `Propose` 注释 "proposals can be lost without notice, therefore it is user's job to ensure proposal retries"（`node.go:138-139`）；`ReadIndex` 同样 "it is user's job to ensure read index retries"（`node.go:222-223`）。持久化顺序（先 Entries/HardState 落盘再发 Messages）也压在宿主身上（`node.go:74-80`、`doc.go:79-82`）。
- **对照 hashicorp/raft**：hashicorp 通过 `NewRaft` 注入 FSM/LogStore/StableStore/SnapshotStore/Transport，库自己起 goroutine 驱动复制、自己调 Transport 发 RPC；etcd/raft 把这些全部退回给宿主，只暴露 Step/Ready。

## 五路径归纳（全部源码核实）

### Hot：Propose → append unstable → 广播 MsgApp → 多数派 → commit → Ready.CommittedEntries
- 提案入口：`Propose`→`MsgProp`（`node.go:471`）→ `run()` 里 `r.Step(m)`（`node.go:389`）。
- Leader 处理 `MsgProp`：`stepLeader`（`raft.go:1275`）case `MsgProp`（:1294）调 `r.appendEntry(m.Entries...)`（:1349）后 `r.bcastAppend()`（:1352）。
- 追加到本地日志：`appendEntry`（`raft.go:812`）为条目盖 `Term/Index`（:818-819），`r.raftLog.append`（:834），并给自己发一条 `MsgAppResp` 自我确认（:845，注释 "The leader needs to self-ack … once durably persisted" :835-838）。
- 提交判定：`MsgAppResp` 成功分支 `pr.MaybeUpdate` → `r.maybeCommit()`（`raft.go:1550`），成功则 `r.bcastAppend()`（:1554）。`maybeCommit`（:775）→ `raftLog.maybeCommit`（`log.go:455`）：`at.term!=0 && at.index>committed && matchTerm` 时 `commitTo`（`log.go:459-461`）。
- 已提交条目出库：`readyWithoutAccept` 填 `CommittedEntries: r.raftLog.nextCommittedEnts(...)`（`rawnode.go:144`）。宿主 apply 后 `Advance()`。

### Control：选举 / tick / becomeX / Step
- 逻辑时钟：`tickElection`（`raft.go:850`，follower/candidate）与 `tickHeartbeat`（`raft.go:862`，leader）由 `RawNode.Tick`（`rawnode.go:64`）→ `r.tick()` 调用。
- 触发选举：`tickElection` 超时发 `MsgHup`（`raft.go:855`）→ `hup`（`raft.go:973`）→ `campaign`（`raft.go:1025`）。
- PreVote：`campaign` 中 `campaignPreElection` 先 `becomePreCandidate`（:1034），`term = r.Term + 1` 但**不真正自增**（:1036-1037）；过了才走 `becomeCandidate`（:1039）真正 `r.Term`。
- 计票：`poll`（`raft.go:1075`）→ `r.trk.RecordVote` + `TallyVotes`（:1081-1082）。
- 状态切换：`becomeFollower`（:891）/`becomeCandidate`（:902）/`becomePreCandidate`（:917）/`becomeLeader`（:933）；`becomeLeader` 追加一条空 entry（:961-965）。
- 单入口分派：`func (r *raft) Step(m)`（`raft.go:1089`）先做 term 比较，再 `r.step(r, m)` 分派到 `stepLeader/stepFollower/stepCandidate`。四态定义 `StateFollower/Candidate/Leader/PreCandidate`（`raft.go:51-54`，`type StateType uint64` :112）。

### State：raftLog + unstable + Storage snapshot
- `raftLog`（`log.go:25-64`）：字段 `storage Storage`（:27）、`unstable`（:31）、三水位 `committed`（:35）、`applying`（:42）、`applied`（:49）；不变式 `applied <= applying && applying <= committed`（:41）。
- `unstable`（`log_unstable.go:37-54`）：暂存尚未落盘的 `entries` 与 `snapshot`，"holds on to new log entries … until they are handed to a Ready struct for persistence"（:24-31）。
- 出/清：`nextUnstableEnts`（`log.go:198`）→ `Ready.Entries`；落盘回执后 `stableTo`（`log.go:367`）→ `unstable.stableTo` 清出。
- 提交/应用水位：`commitTo`（`log.go:322`，never decrease）、`appliedTo`（`log.go:332`）。

### 失败：网络分区 / MsgApp reject 回退 nextIndex
- Follower 侧一致性检查：`handleAppendEntries`（`raft.go:1791`）；`maybeAppend` 失败则回 `Reject:true` + `RejectHint`（`raft.go:1825-1832`），并用 `findConflictByTerm`（:1824）给出跳跃提示。
- Leader 侧回退：`stepLeader` 的 `MsgAppResp` reject 分支（`raft.go:1390`）→ `findConflictByTerm`（:1509）→ `pr.MaybeDecrTo`（:1511）下调 `Next`，`BecomeProbe`（:1514）后 `sendAppend`（:1516）重探。
- 分区/失联：`tickHeartbeat` 里 `MsgCheckQuorum`（`raft.go:869`）；`stepLeader` case `MsgCheckQuorum` 若 `!QuorumActive` 则 `becomeFollower` 退位（`raft.go:1282-1284`）。

### 横切：ReadIndex 线性化读 / joint consensus 成员变更
- ReadIndex：`ReadState{Index,RequestCtx}`（`read_only.go:29-32`）、`readOnly`（:39-47）；`addRequest` 记录当下 commit（:60-62），`recvAck` 收心跳确认（:65-69），`maybeAdvance` 用 `JointConfig.CommittedIndex` 放行（:79-89）。Leader 侧入口 `stepLeader` case `MsgReadIndex`（`raft.go:1354`）：单节点直接回（:1356），本任期未提交则暂存 `pendingReadIndexMessages`（:1365-1367）。宿主经 `Node.ReadIndex`（`node.go:614`）发起。
- 联合共识（joint consensus，etcd/raft 特有，对照 hashicorp 的单步）：`Changer.EnterJoint`（`confchange/confchange.go:51`）进入 `C(new,old)`——注释画出 `(1 2 3)&&()` → `(1 2 3)&&(1 2 3)`（:37-48），对应 Raft 论文 4.3 的 `C_{new,old}`；`LeaveJoint`（:94）收敛到 `C_new`（:86-88）；`Simple`（:128）单步变更、`symdiff > 1` 即报错 "more than one voter changed without entering joint config"（:140-141）。宿主经 `ProposeConfChange`（`node.go:153`，支持 `ConfChangeV2` "arbitrary configuration changes via joint consensus" :148-152）发起，`ApplyConfChange`（`node.go:187`）在 `Ready.CommittedEntries` 里见到 ConfChange 时回调。
- pendingConfIndex 门：`stepLeader` MsgProp 里对 ConfChange 做 `alreadyPending / alreadyJoint / wantsLeaveJoint` 三态校验（`raft.go:1326-1345`），保证同一时刻至多一个未决成员变更。

## 与 hashicorp/raft 的对照要点（供 PANO/arch 用）
| 维度 | etcd-io/raft（本项目） | hashicorp/raft |
|---|---|---|
| 形态 | 共识状态机核，宿主 Node 循环驱动 | 电池全含的可嵌入库 |
| I/O | Ready 外包：宿主做 WAL/网络/线程 | 库注入 Transport/LogStore/... 自驱 |
| 并发 | 库不起驱动复制的 goroutine（RawNode thread-unsafe） | 库单线程 run + 每 follower 一 goroutine |
| 成员变更 | 支持 joint consensus（EnterJoint/LeaveJoint）+ Simple | 单步（一次 ±1） |
| 存储写 | 可选 AsyncStorageWrites（`doc.go:172` 起） | LogStore/StableStore 接口 |
