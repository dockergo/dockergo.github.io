## 判型（system-type verdict）

**多媒体处理框架 · 解复用→解码→滤镜→编码→复用 的可插拔管线 + 统一 AVCodec/AVFormat 抽象。**

- 五段主干是通用骨架，格式/编解码器的多样性收进可插拔组件（demuxer/decoder/filter/encoder/muxer）。
- 上层用一套 `send/receive` 拉推式 API 驱动上千种编解码器；一套 `AVFormatContext` 读写上千种容器。
- 锚点：CLI 转码主循环 `fftools/ffmpeg.c:transcode`（:897），入口 `main`（:991）；n8 起各组件跑在
  `fftools/ffmpeg_sched.c` 的 `Scheduler`（thread-per-component 调度）。

## 设计张力（core tension）

**统一抽象（一套 send/receive API + AVFormat/AVCodec 管上千格式与编解码器）vs 每 codec/容器的特化实现与特化性能路径（硬件加速 / 零拷贝）。**

- 通用性：`avcodec_send_packet`/`avcodec_receive_frame` 对所有解码器同构；`av_read_frame` 对所有 demuxer 同构。
- 特化性：真正干活的是各 codec 的私有实现 + 硬件加速后端（VAAPI/NVDEC/VideoToolbox），帧可留在 GPU
  （`AVHWFramesContext`）走零拷贝，绕开通用 CPU 路径。
- 派生张力：管线通用性 vs 零拷贝（`AVBufferRef` 引用计数浅拷贝，段间不复制像素）。

## 五路径归纳（five-path induction）

### ① Hot · 媒体数据流（一帧/一包的旅程，绿）
- **demux 拆容器** → AVPacket：`libavformat/demux.c:av_read_frame`（:1588；内部 `read_frame_internal`:1388 → `ff_read_packet`:629）。产出压缩数据 + PTS/DTS。
- **decode 解码** → AVFrame：`libavcodec/decode.c:avcodec_send_packet`（:730）配 `libavcodec/avcodec.c:avcodec_receive_frame`（:720；内部 `libavcodec/decode.c:ff_decode_receive_frame_internal`:625）。产出原始像素/采样。
- **filter 滤镜图**（可选）：`libavfilter/avfilter.c:ff_filter_frame`（:1068）；图构建 `libavfilter/avfiltergraph.c:avfilter_graph_alloc`（:85）；两端 `libavfilter/buffersrc.c:av_buffersrc_add_frame_flags`（:210）→ `libavfilter/buffersink.c:av_buffersink_get_frame`（:74）。
- **encode 编码** → AVPacket：`libavcodec/encode.c:avcodec_send_frame`（:544）配 `libavcodec/encode.c:avcodec_receive_packet`（:577；内部 `ff_encode_get_frame`:217）。
- **mux 封容器**：`libavformat/mux.c:av_interleaved_write_frame`（:1223；直写 `av_write_frame`:1176）。按 DTS 交织多流、时间戳 rescale 到输出时基。
- 辅助转换：`libswscale/swscale.c:sws_scale`（:1586，像素格式/缩放）、`libswresample/swresample.c:swr_convert`（:725，重采样）。

### ② Control · 管线编排（谁驱动 send/receive，蓝）
- 转码循环 `fftools/ffmpeg.c:transcode`（:897）编排 demux→dec→filter→enc→mux 的推进；
- send/receive 是「拉推」协议：编排层反复 send 一端、drain 另一端，直到 EAGAIN/EOF。
- n8 调度器 `fftools/ffmpeg_sched.c:Scheduler`：各组件独立线程 + thread_queue 传帧/包。

### ③ State · 缓冲区 refcount（数据载体，琥珀）
- **AVPacket**：`libavcodec/packet.h`（buf :586 / pts :596 / dts :602 / stream_index :605）；引用计数 `libavcodec/packet.c:av_packet_ref`（:442）、`av_packet_make_refcounted`（:497）、`av_packet_unref`（:434）。
- **AVFrame**：`libavutil/frame.h`（data[] :493 / linesize[] :517 / format :559 / pts :574 / buf[] :649）。
- **AVBufferRef 引用计数**（零拷贝核心）：`libavutil/buffer.c:av_buffer_create`（:55）、`av_buffer_ref`（:103，原子 `atomic_fetch_add` :112）、释放 `av_buffer_unref`（原子 `atomic_fetch_sub` 归零才 free :129）。
- **浅拷贝共享缓冲**：`libavutil/frame.c:av_frame_ref`（:278，逐平面 `av_buffer_ref`）、`av_frame_move_ref`（:523）、写时复制 `av_frame_make_writable`（:552）、`av_frame_unref`（:496）。
- 时基载体：AVStream / AVCodecContext 承 `time_base`，PTS/DTS 在段间 rescale。

### ④ 失败降级 / 背压（橙）
- **send/receive 背压协议**：编码端满时 `avcodec_send_frame` 返回 `AVERROR(EAGAIN)`（`libavcodec/encode.c`:225）→ 编排层必须先 drain `avcodec_receive_packet`；解码端同构（`libavcodec/decode.c`:271/280）。EAGAIN 不是错误，是「先取走再喂」。
- **flush/EOF**：send NULL 触发 drain，receive 返回 `AVERROR_EOF`（decode.c:260、encode.c:222）。
- **stream copy 旁路**：`-c copy` 走 `fftools/ffmpeg_mux.c:of_streamcopy`（:280），跳过 decode/encode，直接 demux→mux 重封装（时间戳 `av_rescale_delta`）。

### ⑤ 硬件加速横切（purple，切穿 decode/encode/frame）
- **GPU 帧上下文**：`libavutil/hwcontext.c:av_hwframe_ctx_alloc`（:263）；`AVHWFramesContext` 让 AVFrame 承载 GPU 显存句柄，段间零拷贝。
- **显存↔内存搬运**：`libavutil/hwcontext.c:av_hwframe_transfer_data`（:448）。
- **解码侧接线**：`libavcodec/decode.c:ff_decode_get_hw_frames_ctx`（:1069）、hwaccel 后处理 `fdd->hwaccel_priv_post_process`（:707）。
- 后端：VAAPI / NVDEC(CUDA) / VideoToolbox / QSV / Vulkan。
